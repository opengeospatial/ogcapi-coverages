== Examples (Informative)

This annex demonstrates requests and responses for the resource paths defined by this OGC API - Coverages standard.
For clarity, the multi-dimensional examples within this section are encoded using the https://covjson.org[CoverageJSON] format, which is described by the
https://www.opengis.net/doc/CS/covjson/1.0[OGC CoverageJSON Community Standard].
While CoverageJSON is utilized here to illustrate structured coverage data output, this OGC API - Coverages standard is completely decoupled and agnostic from any single data encoding.
Implementations are required to support content negotiation as part of the Core requirements class.
Servers are free to support other data formats — such as NetCDF, GeoTIFF (including Cloud Optimized with support for HTTP range requests),
JPEG-XL, or LAZ (for point clouds) — based on client request negotiation, and no single data format is universally mandatory.
Supporting binary encodings will provide more efficient bandwidth utilization and processing performance.
https://ubjson.org/[Universal Binary JSON] can be considered as a simple mechanism to turn JSON data into a more efficient binary payload.

=== Common OGC API Resources

==== Landing Page

The landing page provides the main entry point to the API, advertising conformance classes and routing clients
to the collections catalog through explicit link relations.

===== Request

A client requests the top-level API landing page.

[source,http]
----
GET /ogcapi/ HTTP/1.1
Host: example.com
Accept: application/json
----

====== Response

The server returns the landing page payload. The metadata block contains a `links` array that routes clients to the collections list.

[source,http]
----
HTTP/1.1 200 OK
Content-Type: application/json
----

==== Collection Description (Domain)

This example demonstrates how a multi-dimensional coverage collection describes its domain structure within the collection description.
The dataset represents a 4D atmospheric datacube with spatial axes, a temporal axis, and an atmospheric pressure coordinate axis.

The example incorporates explicit grid topology descriptions, including `relativeBounds` for regular grid dimensions (representing the cell-centered offset distance to the
cell edge) and `boundsCoordinates` for irregular grid intervals. For the spatial longitude axis, the 2049 cells at a 0.17578125-degree resolution span exactly 360.0 degrees
from -180.0 to 180.0, indicating a _Value-Is-Point_ grid topology where the outer boundaries represent wrapping point coordinates. Consequently, the spatial `relativeBounds`
collapse to zero. The temporal grid stores instantaneous snapshots, represented by a zero-length duration. For the irregular pressure dimension, `boundsCoordinates` is
structured as an array of arrays of 2 values to define the absolute lower and upper bounds for each pressure level; because these represent exact point-in-space measurements,
the bounds collapse to match the exact coordinate of each level.

==== Schema Resource (Range Type)

This example illustrates how a multi-dimensional coverage collection describes its range type using a standard schema resource per OGC API - Common - Part 3:
Schemas and the Coverages Core requirements class. The resource presents a logical schema defining the fields of the range associated with each coverage cell.
While serialized as a JSON Schema, it serves as a logical description of the data properties rather than a payload validator, ensuring compatibility across
both JSON-based and binary multi-dimensional encodings.

The location of the schema resource is not rigidly fixed to a specific path structure. A client programmatically discovers the correct endpoint at runtime by
inspecting the collection description and resolving the target URL populated within the link object containing the schema link relation type.

===== Request

A client fetches the dynamically discovered schema definition associated with the collection to programmatically determine the structural properties,
sequential order, semantic definition, variable classification type, and statistical boundaries of the internal cell variables before initiating a data stream.

[source,http]
----
GET /ogcapi/collections/climate:era5:relativeHumidity/schema HTTP/1.1
Host: example.com
Accept: application/schema+json
----

===== Response

The server returns a logical JSON Schema document profiling the relative humidity field variable using the key `rh`. It populates standard OGC semantic
annotations to explicitly detail the sequential structure, semantic definition link, percentage unit in UCUM (the default unit language), continuous variable type, and observed
value statistics.

[source,json]
----
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://example.com/ogcapi/collections/climate:era5:relativeHumidity/schema",
  "title": "ERA5 Relative Humidity Range Type Schema",
  "type": "object",
  "properties": {
    "rh": {
      "title": "Relative Humidity",
      "type": "number",
      "description": "The ratio of the partial pressure of water vapor to the equilibrium saturation vapor pressure of water, calculated with respect to the ambient air temperature",
      "x-ogc-propertySeq": 1,
      "x-ogc-definition": "https://qudt.org/vocab/quantitykind/RelativeHumidity",

=== Core Data Request

This example demonstrates baseline data retrieval requests targeting the coverage endpoint without explicit subsetting or field selection parameters.
Under the Core requirements class, if a client requests the entire multi-dimensional dataset without bounds, the server can automatically apply
downsampling based on configured server <<server-limits-recommendations, limits>>. Alternatively, if a client explicitly forbids downsampling via a strict
handling preference, the server rejects requests that exceed native data limits with an HTTP 413 error.

==== Automatic Server-Side Downsampling

When a request for the complete coverage volume is received without a strict handling preference header, a server may automatically scale the
returned array resolution to fit within its internal data transmission safety thresholds.

===== Request

A client requests the full extent of the coverage data cube, negotiating a CoverageJSON response without indicating strict handling constraints.

[source,http]
----
GET /ogcapi/collections/climate:era5:relativeHumidity/coverage HTTP/1.1
Host: example.com
Accept: application/vnd.cov+json
----

===== Response

The server returns a `200 OK` status along with an automatically downsampled CoverageJSON payload. Per the spatial-temporal reference and interpolation
reporting rules, the server populates response headers detailing the exact domain envelope, active reference coordinates, and applied scaling algorithms.

[source,http]
----
HTTP/1.1 200 OK
Content-Type: application/vnd.cov+json
Content-Crs: http://www.opengis.net/def/crs/OGC/1.3/CRS84
Content-Bbox: -180.0,-90.0,180.0,90.0
Content-Datetime: 2023-04-01T00:00:00Z/2023-04-01T12:00:00Z
Content-Interpolation: [ogc-interpolation:linear]
----

==== Strict Handling Enforced (Exceeding Server Limits)

If a client demands the whole coverage extraction at native resolution but fails to provide a `Range:` header, a bounding box
(`bbox`), a temporal interval (`datetime`), or any dimensionality slicing parameters, the server will reject the request if the
native payload size breaks safe thresholds.

===== Request

A client requests the full extent of the coverage data cube but explicitly appends the strict handling header to forbid
automatic server downsampling.

[source,http]
----
GET /ogcapi/collections/climate:era5:relativeHumidity/coverage HTTP/1.1
Host: example.com
Prefer: handling=strict
Accept: application/vnd.cov+json
----

===== Response

Because the strict directive prevents the server from downsampling the dense 4D array to a safe transmission volume, the server
aborts the request and returns an HTTP `413 Content Too Large` error status.

[source,http]
----
HTTP/1.1 413 Content Too Large
Content-Type: application/problem+json

{
  "type": "http://example.com/prob/content-too-large",
  "title": "Payload Too Large",
  "status": 413,
  "detail": "The requested native coverage volume exceeds server data limits. Apply axis subsetting or accept automatic downsampling by omitting the strict handling header."
}
----

=== Axis Subsetting (Slicing and Trimming)

This section provides multiple examples of axis subsetting operations, demonstrating how a client uses trimming parameters to
narrow bounding intervals or slicing parameters to lock a dimension to a single coordinate point.

==== Spatial and Temporal Trimming

This example illustrates a complex trimming request where multiple independent dimensions are restricted to smaller bounding
intervals simultaneously. The client narrows the horizontal spatial footprint and the temporal window to a precise localized box,
while keeping the atmospheric pressure domain fully intact.

A client can execute this trimming operation using global query parameters (`bbox` and `datetime`) or axis-specific `subset`
expressions. Both methods are functionally equivalent and yield the exact same 4D spatial-temporal data slice from the server.

===== Request

[source,http]
----
GET /ogcapi/collections/climate:era5:relativeHumidity/coverage?bbox=10,40,20,50&datetime=2023-04-01T00:00:00Z/2023-04-02T00:00:00Z HTTP/1.1
Host: example.com
Accept: application/vnd.cov+json
----

or equivalently:

[source,http]
----
GET /ogcapi/collections/climate:era5:relativeHumidity/coverage?subset=Lat(40:50),Lon(10:20),time("2023-04-01T00:00:00Z":"2023-04-02T00:00:00Z") HTTP/1.1
Host: example.com
Accept: application/vnd.cov+json
----

===== Response

The server trims the coverage on the latitude, longitude and temporal dimensions, preserving all four dimensions, and returns a `200 OK` status.

[source,http]
----
HTTP/1.1 200 OK
Content-Type: application/vnd.cov+json
Content-Crs: http://www.opengis.net/def/crs/OGC/1.3/CRS84
Content-Bbox: 10.0,40.0,20.0,50.0
Content-Datetime: 2023-04-01T00:00:00Z/2023-04-02T00:00:00Z
Content-Interpolation: [ogc-interpolation:nearest-neighbor]
----

=== Position Query

The Position Query requirements class specifies the ability to query the value(s) for a specific direct position by slicing on every dimension of the coverage, therefore
reducing the dimensionality of the response to zero. Information about the domain or range, other metadata, as well as complex format structures are not necessary and can
negatively affect the bandwidth usage and performance for very large numbers of these queries. This requirements class introduces a simple JSON representation which needs to be
supported by the server. Implementations could also implement additional representations, for example GeoJSON.

==== Mandated Simple JSON Representation

The simple JSON serialization format is mandated when all dimensions are sliced. When a single field is requested for a targeted position that intersects a single cell,
the raw primitive value is returned directly. When multiple fields are requested for that same intersected cell, the server delivers a flat key-value object mapping each field
name to its corresponding value. If the queried position intersects multiple cells, the response is an array containing those values or objects.

===== Returning Single Field

A client targets a digital elevation model (`dem`) collection, slicing across all spatial axes down to an absolute geographic node coordinate position.

====== Request

[source,http]
----
GET /ogcapi/collections/dem/coverage?subset=Lat(45.4844),Lon(-75.72205) HTTP/1.1
Host: example.com
Accept: application/json
----

====== Response

The server locates the exact coordinate point intercept. Because a single field is returned, the payload strips out all dictionary object wrappers and provides the raw numeric elevation scalar token directly.

[source,http]
----
HTTP/1.1 200 OK
Content-Type: application/json
----

[source,json]
----
68.869
----

===== Returning Multiple Fields

A client targets an ERA5 atmospheric data collection (`climate:era5`) that combines multiple multi-dimensional fields.
The query uses explicit subset parameters to slice down the temporal axis, atmospheric pressure layer, and spatial coordinate framework simultaneously to retrieve concurrent
properties.

====== Request

[source,http]
----
GET /ogcapi/collections/climate:era5/coverage?subset=Lat(45.4844),Lon(-75.72205),time("2026-07-03T04:00:00Z"),pressure(850) HTTP/1.1
Host: example.com
Accept: application/json
----

====== Response

The server collapses the multi-dimensional layout down to zero dimensions, packaging the concurrent field values as a single flat key-value object matching the intersected cell.

[source,http]
----
HTTP/1.1 200 OK
Content-Type: application/json
----

[source,json]
----

  "relativeHumidity": 65.2,
  "temperature": 14.8
}
----

=== Interpolation Preferences

This section provides multiple examples demonstrating how a client uses the standard HTTP `Prefer:` header to control server-side
interpolation. These examples cover the different levels of granularity supported by the standard, including global default fallbacks,
per-dimension preferences, per-field overrides, and explicit field-dimension intersections.

==== Global Default Fallback with Per-Dimension Overrides

A client targets the ERA5 collection to downsample the multi-dimensional cell layout using a resolution parameter. The client
passes a `Prefer:` header to establish a global linear interpolation baseline for all fields and dimensions, while explicitly adding
a per-dimension override to enforce nearest-neighbor mapping across the primary temporal dimension.

===== Request

A client issues a scaling query combining a global fallback directive with a targeted temporal axis override inside the HTTP header.

[source,http]
----
GET /ogcapi/collections/climate:era5/coverage?resolution=Lat(90),Lon(180),time(PT12H) HTTP/1.1
Host: example.com
Prefer: interpolation=[ogc-interpolation:linear], interpolation=time([ogc-interpolation:nearest-neighbor])
Accept: application/vnd.cov+json
----

===== Response

The server honors the global baseline and the dimension-specific override, downsamples the fields, and returns a `200 OK` status. The response
headers include the `Preference-Applied:` token to confirm compliance alongside the compact `Content-Interpolation:` token to report the applied logic.

[source,http]
----
HTTP/1.1 200 OK
Content-Type: application/vnd.cov+json
Preference-Applied: interpolation=[ogc-interpolation:linear], interpolation=time([ogc-interpolation:nearest-neighbor])
Content-Interpolation: [ogc-interpolation:linear]; dimensions=time([ogc-interpolation:nearest-neighbor])
----
_[Data Payload Omitted]_

==== Per-Field Interpolation Override

This example illustrates a field-specific query targeting a mixed environmental dataset (`climate:landuse`). The coverage carries
a continuous field (`temperature`) and a non-continuous categorical field (`landCover`). While a server following standard
recommendations should automatically select nearest-neighbor logic for categorical fields to prevent value corruption, a client
can be proactive by explicitly requesting nearest-neighbor interpolation exclusively on the `landCover` field.

===== Request

A client uses the `properties-interpolation` parameter modifier inside the `Prefer:` header string to explicitly target a field.

[source,http]
----
GET /ogcapi/collections/climate:landuse/coverage?resolution=Lat(90),Lon(180) HTTP/1.1
Host: example.com
Prefer: properties-interpolation=landCover([ogc-interpolation:nearest-neighbor])
Accept: application/vnd.cov+json
----

===== Response

The server confirms the preference for the land cover field while maintaining its continuous default linear interpolation logic for the
unspecified temperature field. Because the respective methods are uniform across all dimensions within each field, the server leverages
global field shorthand notation to deliver a condensed response header.

[source,http]
----
HTTP/1.1 200 OK
Content-Type: application/vnd.cov+json
Preference-Applied: properties-interpolation=landCover([ogc-interpolation:nearest-neighbor])
Content-Interpolation: [ogc-interpolation:linear]; properties=landCover([ogc-interpolation:nearest-neighbor])
----
_[Data Payload Omitted]_

but demands that the `temperature` field use exact linear interpolation specifically for the spatial `Lat` and `Lon` dimensions
and the primary temporal dimension, but explicitly forbids interpolation along the atmospheric pressure axis.

[source,http]

==== Client error under mutually exclusive selection parameters

This example demonstrates the error handling behavior when a client mixes conflicting field selection query parameters. Per the specification
constraints, inclusive parameter listing and exclusive parameter listing are mutually exclusive options.

===== Request

A client issues an invalid query attempting to simultaneously leverage both inclusive and exclusive selectors inside the same request string.

[source,http]
----
GET /ogcapi/collections/climate:era5/coverage?properties=temperature&exclude-properties=relativeHumidity HTTP/1.1
Host: example.com
Accept: application/json
----

===== Response

The server detects the co-occurrence of both parameters, aborts processing, and returns a `400 Bad Request` status code accompanied
by a standard JSON error message payload.

[source,http]
----
HTTP/1.1 400 Bad Request
Content-Type: application/problem+json
----

[source,json]
----
{
  "type": "http://www.opengis.net/def/exceptions/ogcapi-coverages-1/1.0/conflicting-parameters",
  "title": "Invalid Field Selection Parameters",
  "status": 400,
  "detail": "The 'properties' and 'exclude-properties' query parameters are mutually exclusive and cannot be combined in a single request."
}

=== Output CRS selection

The Coordinate Reference System (CRS) Selection requirements class provides the ability to request a coverage in a specific target CRS instead of the collection's native CRS. The available target options are advertised to the client within the `crs` array of the collection description. A client uses the `crs` query parameter to specify a target CRS identifier chosen from this supported list.

==== Spatial reprojection to a projected CRS

This example illustrates a request against our 4D atmospheric `climate:era5` collection, which has a geographic `CRS84` coordinate reference system as its native CRS. A client requests the data to be reprojected into the World Mercator coordinate reference system (`EPSG:3395`) to obtain conformal projected metric coordinates for navigational modeling.

===== Request

A client targets the coverage endpoint, passing a supported target coordinate reference system identifier inside the `crs` query parameter.

[source,http]

Host: example.com
Accept: application/vnd.cov+json
----

HTTP/1.1 200 OK
Content-Type: application/vnd.cov+json
Content-Crs: http://www.opengis.net/def/crs/EPSG/0/3395
Content-Bbox: 1113194.0,5621521.0,2226389.0,6446275.0

=== Tiled Coverage Data

The Tiled Coverage Data requirements class bridges the OGC API - Coverages standard with the OGC API - Tiles and OGC 2D Tile Matrix
Set (2DTMS) standards. This class allows a client to fetch pre-aligned blocks of cells by targeting specific matrix, row,
and column coordinates defined by a 2DTMS, rather than submitting arbitrary spatial subset bounding boxes.

A request for tiled coverage data is equivalent to a specific spatial subset operation defined by the boundaries of the requested tile.
The response payload structures the dimensions to align with the selected tiling scheme, while either including or slicing additional
dimensions like time or pressure.

dimensions are collapsed to a single coordinate point using query parameters.

and column. This expands the production grid shape from the standard 256x256 layout to a 257x257 array footprint containing exactly

==== Retrieving multi-dimensional tiled coverage data

This example demonstrates retrieving multi-dimensional tiled coverage data from a World Mercator 2D Tile Matrix Set, without slicing on the other non-spatial dimensions.
The server delivers a complete 4D tiled data payload containing two time steps and two pressure layers, producing a spatial sub-matrix for each coordinate combination.

For the projected 2D Tile Matrix Set like _WorldMercatorWGS84Quad_ (`EPSG:3395`), the coordinates are expressed in metric meters.
For the example collection, cells are defined as point-centered observations (_Value-is-Point_).
For the official tiling scheme defines a standard matrix envelope of 256x256 cells, the space partition for point-centered data natively requires an extra bounding edge node
row and column, expanding the production grid shape to a 257x257 array footprint per coordinate combination.

[NOTE]
For the purpose of a shorter example to include in this annex, the response payload shown below is truncated to assume a tile matrix width
and height of 16x16 cells instead of the native 256x256 dimensions. Due to the point-centered grid structure, adding the bounding edge node
row and column results in a complete 17x17 spatial array footprint. The full 4D matrix block populates a total array size of 1,156 values
per field (2 times x 2 pressure levels x 17 rows x 17 columns).

===== Request

The client requests a tile from the Mercator ERA5 tileset, without any subsetting query parameter therefore preserving the temporal and pressure dimensions within the tile data.

[source,http]
----
GET /ogcapi/collections/climate:era5/tiles/WorldMercatorWGS84Quad/2/1/2 HTTP/1.1
Host: example.com
Accept: application/vnd.cov+json
----
