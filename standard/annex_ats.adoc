|requirement: |/req/core/collection-description
|test purpose: |Verify the coverage properties, links, and coordinate reference systems in collection descriptions.
|test method: |
*Given:* An API implementation claiming conformance to the OGC API - Coverages - Part 1: Core standard +
*When:* Collection descriptions are retrieved from the API's collection resources +
*Then:* +
- Assert that each collection accessible using the Coverages API includes a link in its `links` array targeting a coverage resource with the link relation type `[ogc-rel:coverage]` (or its equivalent full URI). +
- Assert that at least one collection is accessible as a coverage. +
- Assert that, if a `storageCrs` property is present, it contains a valid URI, safe CURIE, WKT2, PROJJSON, or CRS-JSON object describing a registered or full CRS definition. +
- Assert that, if a `storageCrs` property is present and its value is not the standard CRS84 URI, a `storageCrsBbox` property is provided within the spatial dimension of the `extent` object adhering to the specified bounding box schema.

**Abstract Test A.{counter:ats-id}:** Get Coverage
[cols=">20h,<80d",width="100%"]
|===
|test id: |/conf/core/get-coverage
|requirement: |/req/core/get-coverage
|test purpose: |Verify the operational execution, status codes, content negotiation, coordinate reference systems, and field ordering of the coverage retrieval endpoint.
|test method: |
*Given:* An API implementation claiming conformance to the OGC API - Coverages - Part 1: Core standard +
*When:* A coverage resource is requested via an HTTP GET operation using the endpoint identified by the `[ogc-rel:coverage]` link relation type in the collection description +
*Then:* +
- Assert that a successful execution returns an HTTP status code `200`. +
- Assert that the response body contains the coverage data and utilizes the self-description capabilities of the negotiated content type. +
- Assert that the response is encoded using the format negotiated through the HTTP protocol. +
- Assert that, if no format is negotiated and a `type` property is present in the link object within the collection description resource, the response is encoded using the media type format specified by that property. +
- Assert that the coverage response is in the native CRS (`storageCrs`) specified in the collection description, or the standard CRS84 URI if none is specified, unless overridden by a specific query parameter. +
- Assert that, if the response format uses a concept of field order (e.g., GeoTIFF), that order matches the requested fields based on explicit selection, `defaultFields` in the collection description, or the `x-ogc-propertySeq` sequence in the collection schema.

|===
|test id: |/conf/core/response-headers
|requirement: |/req/core/response-headers
|test purpose: |Verify the coordinate reference system, spatial bounding box, and temporal information provided in the coverage response headers.
|test method: |
*Given:* An API implementation claiming conformance to the OGC API - Coverages - Part 1: Core standard +
*When:* A coverage resource is retrieved from the API +
*Then:* +
- Assert that, if the content is referenced to a CRS other than CRS84, the response includes a `Content-Crs:` header containing the URI of the response CRS. +
- Assert that, if the returned coverage has a spatial extent, the response includes a `Content-Bbox:` header containing the actual spatial extent of the coverage. +
- Assert that, when present, the `Content-Bbox:` coordinates are expressed in the response CRS, formatted as four or six comma-separated numbers representing the lower-left and upper-right corners, and honor the coordinate axis order of that CRS. +
- Assert that, if the coverage includes a temporal dimension expressible in RFC3339 format, the response includes a `Content-Datetime:` header containing either a single datetime instant for a single temporal slice, or a datetime interval when the response spans a temporal dimension or contains data originating from multiple aggregated or mosaiced temporal slices.

header to be missing only if the interpolation remains constant at the server's default method rather than reflecting what the client requested.

**Abstract Test A.{counter:ats-id}:** Spatial Subset Parameter
[cols=">20h,<80d",width="100%"]
|===
|test id: |/conf/subsetting-spatial/subset
|requirement: |/req/subsetting-spatial/subset
|test purpose: |Verify the baseline slicing, trimming, axis mappings, and coordinate interpretations of the spatial `subset` query parameter.
|test method: |
*Given:* An API implementation claiming conformance to the OGC API - Coverages - Spatial Subsetting requirements class +
*When:* A coverage resource request is executed using a valid single coordinate or interval expression targeting spatial dimensions matching the defined ABNF grammar +
*Then:* +
- Assert that the implementation parses and accepts valid `subset` expressions for both intervals and discrete coordinates across one or more dimensions. +
- Assert that the returned coverage data includes only those constituent cells whose interiors intersect the interior of the specified subsetting coordinates or intervals. +
- Assert that a single coordinate subset on an axis executes a slicing operation that reduces the dimensionality of the response. +
- Assert that an interval subset on an axis executes a trimming operation that preserves the dimensionality of the response. +
- Assert that axis names `Lat` and `Lon` are supported for geographic subsetting CRSs and map to the corresponding spatial axes. +
- Assert that axis names `E` and `N` are supported for projected subsetting CRSs, and axis names `x` and `y` are supported for 3D Cartesian CRSs. +
- Assert that spatial subset coordinates are interpreted using the CRS specified by the `subset-crs` parameter, defaulting to `[OGC:CRS84]` (or `CRS84h`) if the parameter is absent.
